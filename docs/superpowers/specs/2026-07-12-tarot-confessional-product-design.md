# 塔罗树洞 Agent Skill 产品设计

- 日期：2026-07-12
- 状态：已确认方向，待实现
- 版本范围：v0.1 互动塔罗，v0.2 本地记忆与树洞
- 关联项目：同级目录 `MNEME`

## 1. 产品定义

塔罗树洞不是一个独立占卜网站，也不是一个只会输出牌义的提示词。它是一个由 Agent 主持、浏览器承担视觉交互、轻量脚本保障确定性、LLM 完成理解与综合解读的 Agent Skill。

用户始终从对话开始，也回到对话完成核心交流。HTML 是 Agent 在关键节点递给用户的“体验器皿”：第一张负责抽牌，第二张负责承载最终解读。

### 1.1 用户价值

1. 对话让用户能说清真正关心的事，而不是从固定问题列表中选择。
2. 抽牌页提供实体牌桌般的仪式感、随机性和视觉反馈。
3. Agent 将牌义、牌位和用户上下文综合起来，避免机械拼接关键词。
4. 报告页把一次易消失的聊天变成可阅读、打印和保存的个人文稿。
5. v0.2 在明确授权后使用本地记忆，帮助用户观察长期主题，而不是制造依赖。

### 1.2 非目标

- 不宣称预知确定未来。
- 不替代医疗、心理咨询、法律、财务或紧急援助。
- 不要求云端账号、后端服务或联网抽牌。
- 不在 v0.1 保存日记、问题或解读历史。
- 不把固定牌义直接拼成所谓“AI 解读”。
- 不建立情绪评分、人格诊断或风险画像。

## 2. 版本路线

### v0.1：互动塔罗闭环

完成“对话澄清 -> 打开抽牌页 -> 返回抽牌码 -> Agent 解码和解读 -> 打开报告页”的完整体验。

成功标准：用户不需要安装网页应用，不需要联网，不需要手工输入牌名，也能得到一次可信、漂亮且可复现的完整体验。

### v0.2：树洞、日记与本地记忆

在 v0.1 之上增加可选的 MNEME 适配层。用户可以主动保存一次阅读、写入日记、检索过去主题，并通过 dream 生成周期性整理。

成功标准：记忆默认关闭、保存前确认、原始内容本地可见可删，任何长期引导都有来源并避免心理诊断。

## 3. v0.1 端到端流程

```text
用户发起塔罗请求
       |
       v
Agent 澄清诉求、选择牌阵、检查高风险边界
       |
       v
Agent 生成并交付 draw.html
       |
       v
浏览器本地洗牌和抽牌，生成抽牌码
       |
       v
用户将抽牌码粘贴回对话
       |
       v
Agent 调用 decode_draw.py 校验并还原牌面
       |
       v
Agent 按 references/ 中的牌义和解读框架综合分析
       |
       v
Agent 调用 render_report.py 生成 reading.html
       |
       v
用户阅读、打印或保存报告
```

### 3.1 对话阶段

Agent 首先判断用户属于以下哪类诉求：

- 聚焦问题：例如关系走向、工作选择、近期状态。
- 开放探索：用户只有模糊感受，希望抽牌帮助整理。
- 纯树洞：用户想先说，不一定需要抽牌。
- 高风险请求：涉及伤害、疾病诊断、死亡、违法或重大财务决策。

普通场景最多追问一个核心问题，避免把仪式拖成问卷。建议询问：“你更想看清现状、下一步，还是一段关系中的双方？”

高风险场景先执行安全边界；必要时不进入抽牌流程。

### 3.2 牌阵选择

v0.1 只提供足够覆盖常见场景的三种牌阵：

| ID | 牌阵 | 牌位 | 适用场景 |
|---|---|---|---|
| `F1` | 一张牌 | 此刻的提醒 | 日签、聚焦提示 |
| `S3` | 三张行动牌阵 | 现状、阻力、可行动方向 | 选择、工作、一般困惑 |
| `R3` | 三张关系牌阵 | 自己、对方、关系动力 | 关系与沟通 |

v0.1 不支持十字等大型牌阵。限制牌数可以提高移动端体验、报告质量和视觉资产可控性。

### 3.3 抽牌页生成

Agent 调用确定性脚本，而不是现场手写 HTML：

```bash
python3 scripts/create_draw_page.py \
  --spread S3 \
  --title "关于下一步的三张牌" \
  --output <workspace>/tarot-draw.html
```

脚本把配置、安全转义后的标题、牌库版本和所需资源嵌入自包含 HTML。问题正文不写入 HTML，避免在浏览器文件中留下敏感信息。

### 3.4 浏览器抽牌

- 使用 `crypto.getRandomValues()` 洗牌，无可用实现时明确报错，不降级到 `Math.random()`。
- 从 78 张完整牌组中无放回抽取。
- 每张牌独立决定正位或逆位，默认概率各 50%。
- 牌在用户主动选择前保持未知。
- 页面不请求网络、不写 cookie、不使用 analytics、不读取剪贴板。
- 只有用户点击“复制抽牌码”时才写剪贴板，并提供手动选择的回退方式。

### 3.5 抽牌码协议

抽牌码是浏览器与 Agent 之间的传输协议，不是加密或防作弊机制。

格式：

```text
TC1-<SPREAD>-<PAYLOAD>-<CHECKSUM>
```

示例：

```text
TC1-S3-7K4M2Q9D-5H
```

设计约束：

- `TC1` 是协议版本，未来升级不会误解旧码。
- `SPREAD` 是牌阵 ID。
- `PAYLOAD` 使用排除 `0/O/1/I/L` 的 Crockford 风格 Base32，编码牌 ID、正逆位和顺序。
- `CHECKSUM` 使用 CRC-10 或等价的短校验，发现复制错误。
- 不编码用户问题、时间、设备、身份或任何隐私数据。
- 解码器必须拒绝未知版本、未知牌阵、重复牌、越界牌 ID 和错误校验码。

编码算法同时存在于抽牌页脚本和 Python 解码器中，并通过共享测试向量保证一致。Agent 不应凭语言模型推测抽牌码。

### 3.6 牌库与牌义数据

牌面事实和解释参考分开存储：

```text
references/
├── deck.json                 # ID、牌名、组别、数字、图像键
├── meanings/
│   ├── major-arcana.md       # 大阿卡纳参考
│   ├── wands.md
│   ├── cups.md
│   ├── swords.md
│   └── pentacles.md
├── spreads.md                # 牌阵和牌位关系
├── interpretation-guide.md   # 综合解读方法
└── safety.md                 # 高风险边界与响应方式
```

`deck.json` 是解码事实来源；Markdown 参考是给 Agent 的语义材料。每张牌提供核心主题、光明面、阴影面、正位倾向、逆位倾向和反思问题，但不提供可直接复制的固定结论。

### 3.7 Agent 解读顺序

1. 重述问题的核心张力，不复述不必要的隐私细节。
2. 说明这是象征性反思，不是确定性判断。
3. 先解释牌位，再解释牌，不把每张牌孤立成词典条目。
4. 识别牌之间的重复元素、数字、方向和矛盾。
5. 把解读连接回用户已提供的现实信息。
6. 给出一到两个可执行但低风险的行动或反思问题。
7. 避免“宇宙告诉你”“一定会”“命中注定”等权威语言。

### 3.8 报告页生成

Agent 先形成结构化 JSON，再交给脚本渲染：

```json
{
  "schema_version": "1",
  "spread_id": "S3",
  "title": "关于下一步的三张牌",
  "question_summary": "如何判断现在是否适合改变工作方向",
  "cards": [
    {
      "position": "现状",
      "card_id": 0,
      "orientation": "upright",
      "reading": "..."
    }
  ],
  "synthesis": "...",
  "reflection_prompts": ["..."],
  "boundary_note": "..."
}
```

```bash
python3 scripts/render_report.py \
  --input <workspace>/reading.json \
  --output <workspace>/tarot-reading.html
```

渲染器负责转义所有文本、校验 schema、嵌入对应牌图、生成打印样式。Agent 不直接拼接用户文本到 HTML，避免 HTML 注入和格式漂移。

### 4.1 分层职责

| 层 | 负责 | 不负责 |
|---|---|---|
| Agent 对话 | 理解意图、选择牌阵、综合解释、安全判断 | 随机数、码解析、HTML 转义 |
| Python 脚本 | 编解码、数据校验、模板渲染 | 替用户作心理判断 |
| HTML 抽牌页 | 视觉体验、本地随机、生成抽牌码 | 解释牌义、保存问题 |
| HTML 报告页 | 展示、打印、保存 | 二次推理、网络上传 |
| references | 牌面事实、解释框架、安全规范 | 固定输出结论 |

## 5. 隐私与安全

- v0.1 全流程不需要网络请求。
- 抽牌页只包含牌阵和非敏感标题，不包含问题正文。
- 报告页包含用户内容，因此 Agent 生成前应提醒它会写入本地文件。
- 文件名不包含问题、姓名、关系对象或健康信息。
- 不把用户内容写入仓库或 Skill 安装目录，产物写到当前工作区或临时目录。
- 所有牌图、字体和纹理必须有清晰许可证与来源清单。
- 报告中的危机提示应简洁具体，不利用神秘叙事强化恐惧。

## 6. Agent Skill 触发与渐进加载

`SKILL.md` 保持短小，负责：

- 触发：塔罗、抽牌、牌阵、感情占卜、工作选择、树洞、想倾诉、日记回顾。
- 路由：v0.1 塔罗、v0.2 树洞/记忆、高风险安全处理。
- 工具调用：生成抽牌页、解码、生成报告。
- 指向 references：只有实际需要时读取牌义、牌阵、安全或 MNEME 文档。

完整牌义不塞进 `SKILL.md`，防止每次触发都占用大量上下文。

## 7. 失败状态

| 场景 | 行为 |
|---|---|
| 浏览器不支持安全随机 | 页面停止抽牌，提示换现代浏览器 |
| 剪贴板权限失败 | 展示可选择文本与明确复制按钮状态 |
| 抽牌码输错 | 解码器指出格式或校验错误，请用户重新复制 |
| 抽牌码版本未知 | 不猜测，提示使用匹配版本页面重新抽牌 |
| 牌图缺失 | 报告仍生成，使用文字牌面和明确占位，不阻断解读 |
| 用户拒绝生成本地报告 | 在对话中提供纯文本解读 |
| 高风险内容 | 暂停仪式流程，转为现实安全支持 |

## 8. 验收标准

### 功能

- 三种牌阵均可完成抽牌、复制、解码和报告生成。
- 1000 次随机抽牌不出现同一牌阵内重复牌。
- JS 与 Python 对固定测试向量得到完全一致结果。
- 任意单字符抽牌码错误能被校验发现。
- HTML 在断网状态可完整使用。

### 体验

- 首屏直接进入抽牌体验，不是产品介绍页。
- 桌面和 360px 宽移动端无溢出、遮挡或不可点击区域。
- 键盘可以完成选择、确认和复制。
- `prefers-reduced-motion` 下无洗牌、漂浮或翻转动画，但流程完整。
- 报告可打印为清晰的 A4/PDF。

### 安全

- 抽牌码不包含用户问题或身份数据。
- 用户输入经过 schema 校验和 HTML 转义。
- 高风险测试用例不会得到确定性占卜结论。
- 页面不存在网络请求、追踪脚本或持久化存储。

## 9. 关键决策记录

| 决策 | 选择 | 原因 |
|---|---|---|
| 网页形态 | 自包含静态 HTML | 跨 Agent 平台、无服务、可离线 |
| 随机来源 | Web Crypto | 比 `Math.random()` 更可靠且原生 |
| Agent 传输 | 版本化抽牌码 | 人可复制、机器可校验、无隐私 |
| 解码方式 | 确定性脚本 | 不让 LLM 猜协议 |
| v0.1 牌数 | 1 或 3 | 控制体验、资产和报告复杂度 |
| 牌义存储 | JSON 事实 + Markdown 解释 | 机器准确与 Agent 可读兼顾 |
| HTML 安全 | 结构化 JSON + 模板渲染 | 防注入并保证视觉一致 |
| 长期记忆 | v0.2 可选 MNEME | 先证明核心体验，再增加隐私复杂度 |

## 10. 暂不决策

- 项目正式中文名、英文名和 Logo。
- 最终使用公共领域 RWS 图像还是原创完整牌组。
- 逆位是否默认开启，首轮用户测试后决定。
- 是否提供五张以上牌阵。
- v0.2 是否将 MNEME 作为外部依赖、Git 子模块或安装时检测的可选 Skill。
